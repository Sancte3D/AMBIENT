# Rule: AI-Ready PCB Schematic Standard

**Status: BINDING (r18.41, 2026-06-21).** Applies to every create / review /
modify action on any PCB schematic in this repo. Referenced as a hard rule
from the repo-root `CLAUDE.md` so it loads into every session.

When creating, reviewing, or modifying any PCB schematic, always structure it
so that both a human engineer and an AI system can understand it without
guessing.

## Core Rule

The schematic must be treated as **machine-readable design data**, not only as
a visual drawing. Every component, net, sheet, symbol, footprint, and
functional block must be explicit, named consistently, and traceable.

> In THIS project the schematic is generated by
> `kicad/generate_kicad_project.py`. "Modifying the schematic" therefore means
> **editing the generator**, never hand-editing `.kicad_sch`. All standards
> below are enforced in the generator's per-component / per-net code.

## Scope — don't over-detail; nail the buildable core

The goal is a schematic that can actually be **built** and understood, not a
maximally annotated one. Prioritise accordingly:

**MUST-HAVE for every part / net (the schematic is not buildable without these):**
1. **Correct reference designator + descriptive name**
2. **Value** (R/C/L value, or the part's function)
3. **Footprint** (the exact one for the real package)
4. **Connectivity** — what connects to what, with which pin (this is the whole
   point: anyone, human or AI, must be able to read where each net goes)

**WHERE SENSIBLE (add when it matters for that part, skip the noise):**
- Full metadata (MPN, supplier PN, ratings, tolerance, lifecycle, datasheet)
  → essential on ICs / connectors / power parts; minimal on jellybean passives.
- Title-block author/notes, dedicated test points, exhaustive NC annotation
  → useful, not blockers. Add for critical nets, don't drown the sheet.

So: a 10 kΩ 0603 needs name + value + footprint + where its two ends connect —
not a paragraph. An IC or connector needs the full treatment (pinout verified,
package exact, orientation clear).

## Required Standards

### 1. Clean functional blocks
Split the schematic into logical sections, each with a clear title + purpose:
Power Input & Regulation · MCU / Processor Core · Audio / Sensor / Interface ·
Communication Interfaces · Display / UI Controls · Debug / Programming ·
Connectors · Mechanical / Mounting.

### 2. Consistent net naming
- Descriptive names, not random pin names: `3V3_IO`, `5V_USB`, `VBAT`,
  `MCU_RESET_N`, `SPI_MOSI`, `SPI_MISO`, `SPI_SCK`, `DISPLAY_CS`, `ENC_A`,
  `ENC_B`, `AUDIO_L_OUT`, `AUDIO_R_OUT`.
- **`_N` suffix for active-low** (e.g. `AMP_SHDN_N`, not `nSHDN`).
- **`_P` / `_N` for differential pairs.**
- No spaces, slashes, unclear abbreviations, or inconsistent capitalisation.

### 3. Complete component metadata
Each component includes, where possible: Manufacturer Part Number ·
Manufacturer Name · Supplier Part Number (LCSC / DigiKey / Mouser) · Package ·
Footprint · Value · Voltage / current / power rating · Tolerance (where
relevant) · Lifecycle status (if known) · Datasheet link / source reference.

### 4. Verified symbol-to-footprint mapping
Never assume a footprint is correct just because it exists. For every critical
component: check symbol pin numbers vs datasheet; check footprint pad numbers
vs datasheet; check polarity / pin 1 / orientation / notch / diode direction /
connector orientation; check the exact package variant (not "roughly"); if
JLC/LCSC assembly is intended, verify the exact LCSC part and exact footprint
map to the same physical package.

### 5. Standard, unique reference designators
`R` resistor · `C` capacitor · `U` IC · `J` connector · `L` inductor ·
`D` diode · `Q` transistor/MOSFET · `TP` test point · `Y` crystal/oscillator ·
`FB` ferrite bead · `SW`/`S` switch/button. Every designator unique and stable
across revisions.

### 6. Labels instead of long wires
Signal flow left→right; power top→bottom; related parts close together; long
crossing wires replaced with net labels; no visual spaghetti.

### 7. Title blocks + revision context
Every sheet: project name · sheet title · sheet number · revision · date ·
author/reviewer · short change notes if relevant.

### 8. Mark all intentional no-connects
No pin ambiguous. Unused IC pins marked NC only if the datasheet allows.
Unused inputs tied high/low where required. Enable, boot, reset, mode,
address, config pins always explicitly handled.

### 9. Explicit power design
Always show: input voltage range · power rails · regulators · required
input/output caps from datasheet · decoupling caps near ICs · current
capacity · protection circuits where needed · ground domains if analog /
digital / power ground are separated.

### 10. Test + debug access
Useful test points + programming/debug headers: `TP_3V3`, `TP_GND`,
`TP_RESET`, `TP_I2C_SCL`, `TP_I2C_SDA`, `SWDIO`, `SWCLK`, `UART_TX`, `UART_RX`.

## Claude Behavior Rule

When reviewing a schematic, do not only check if it "looks correct." Check if
it is **AI-readable, manufacturable, and verifiable.** For every issue report:
**what is wrong · why it matters · what could fail** (PCB layout, JLC
assembly, firmware, or debugging) **· how to fix it · severity** (critical /
important / minor).

## Do Not Guess

If a footprint, pinout, datasheet, package, polarity, connector orientation,
or supplier part cannot be verified, mark it exactly:

```
UNVERIFIED — NEEDS HUMAN CHECK
```

Never pretend a footprint is safe unless symbol, datasheet, package, and
footprint have been matched directly.
